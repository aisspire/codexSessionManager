# Codex Session Manager 使用说明

这份说明面向日常使用，重点写“怎么操作”和“什么时候该小心”。如果只想了解项目背景和代码结构，请看 `README.md`。

## 1. 使用前准备

### 1.1 找到 Codex 主目录

常见位置：

```text
Windows: C:\Users\<用户名>\.codex
WSL:     /mnt/c/Users/<用户名>/.codex
Linux:   ~/.codex
```

本文示例使用：

```text
/mnt/c/Users/14139/.codex
```

请按自己的机器替换。

### 1.2 写入前关闭 Codex

执行归档、恢复、删除、迁移、重命名、修改 provider 或项目路径前，请关闭：

- Codex Desktop
- Codex CLI
- Codex app-server

工具会在写入前检查 Codex 是否正在运行。检测到运行中会拒绝写入，避免和 Codex 自身同时改本地状态。

### 1.3 先备份

手动备份：

```bash
cargo run -- \
  --codex-home /mnt/c/Users/14139/.codex \
  backup
```

默认备份关键状态文件，不复制完整 `sessions/`。如果要连聊天 JSONL 一起复制：

```bash
cargo run -- \
  --codex-home /mnt/c/Users/14139/.codex \
  backup --include-sessions
```

写入类操作默认也会自动备份关键文件。

## 2. 桌面端使用

### 2.1 启动

```bash
cd ui
npm install
npm run tauri dev
```

如果 WSL 里的 Node 环境不可用，可以从 Windows PowerShell 运行：

```powershell
powershell.exe -NoProfile -ExecutionPolicy Bypass -Command "Set-Location 'E:\code\codexSessionManager\ui'; npm run tauri dev"
```

### 2.2 基本界面

桌面端分为三块：

- 左侧：Codex 主目录、筛选条件、批量修改已选。
- 中间：会话表格。
- 右侧：当前会话详情和单会话操作。

### 2.3 筛选会话

左侧可以按这些字段筛选：

- 项目
- 提供方
- 模型
- 来源
- 搜索关键词
- 全部 / 活动 / 已归档

填好后点 **刷新**。

### 2.4 选择会话

- 点击表格行：打开右侧详情。
- 勾选行首 checkbox：加入多选。
- 点击表头第一列 checkbox：全选当前列表。
- 表头 checkbox 半选：表示当前列表只有一部分被选中。

全选只作用于当前筛选后的可见列表。

### 2.5 修改已选会话

左侧 **修改已选** 面板可以批量修改：

- 会话名前缀
- 提供方
- 项目路径

空字段不会修改。

批量修改步骤：

1. 在表格中勾选一个或多个会话。
2. 在左侧填写要修改的字段。
3. 点 **预览** 查看影响数量。
4. 确认后点 **应用**。
5. 写入前会弹出确认，并自动创建备份。

多选会话名前缀规则：

```text
前缀(1)
前缀(2)
前缀(3)
```

编号按时间从早到晚递增，从 1 开始，没有空格。

例如前缀填 `Review`，会生成：

```text
Review(1)
Review(2)
Review(3)
```

### 2.6 右侧单会话修改

右侧详情支持单独修改：

- 会话名称
- 项目路径
- 提供方

操作方式：

1. 点击字段旁边的小笔。
2. 字段变成输入框。
3. 输入新值。
4. 按回车或点击外侧确认草稿。
5. 底部 **保存** 按钮变为可用。
6. 点击 **保存** 后真正写入。

如果没有修改，保存按钮是灰色不可用。

单会话保存会写入：

- 会话名称：`state_5.sqlite` 的 `threads.title`，以及 `session_index.jsonl` 的 `thread_name`。
- 项目路径：`state_5.sqlite` 的 `threads.cwd`，以及对应 JSONL 第一行 `session_meta.payload.cwd`。
- 提供方：`state_5.sqlite` 的 `threads.model_provider`，以及对应 JSONL 第一行 `session_meta.payload.model_provider`。

### 2.7 归档、恢复、删除

工具栏和右侧详情都提供：

- 归档
- 恢复
- 删除

删除不是永久删除。它会把会话 JSONL 移到：

```text
<CodexHome>/trash/codex-session-manager/<timestamp>/
```

并写入 `manifest.json`，同时把 SQLite 中对应线程标记为 archived。

## 3. CLI 使用

CLI 命令格式：

```bash
cargo run -- \
  --codex-home /mnt/c/Users/14139/.codex \
  <command>
```

如果已经构建二进制：

```bash
./target/debug/codex-session-manager \
  --codex-home /mnt/c/Users/14139/.codex \
  <command>
```

### 3.1 扫描

```bash
cargo run -- \
  --codex-home /mnt/c/Users/14139/.codex \
  scan
```

重点看：

```text
threads with non-active provider
rollout_path values matching path maps
cwd values matching path maps
missing rollout files
JSONL/SQLite id mismatches
visible user threads missing from session_index
```

### 3.2 列出会话

```bash
cargo run -- \
  --codex-home /mnt/c/Users/14139/.codex \
  list --archived all
```

筛选示例：

```bash
cargo run -- \
  --codex-home /mnt/c/Users/14139/.codex \
  list --project /mnt/e/code/demo --provider cm --archived active
```

### 3.3 迁移 provider

先 dry-run：

```bash
cargo run -- \
  --codex-home /mnt/c/Users/14139/.codex \
  migrate-provider --from openai --to cm
```

确认数量后应用：

```bash
cargo run -- \
  --codex-home /mnt/c/Users/14139/.codex \
  migrate-provider --from openai --to cm --apply
```

它会修改：

- `state_5.sqlite` 的 `threads.model_provider`
- `sessions/**/*.jsonl` 第一行 `session_meta.payload.model_provider`

不会修改历史聊天正文。

### 3.4 迁移路径

从 Windows 路径迁移到 WSL 路径时，用 `--path-map FROM=TO`。

先 dry-run：

```bash
cargo run -- \
  --codex-home /mnt/c/Users/14139/.codex \
  --path-map 'E:\code=/mnt/e/code' \
  migrate-paths
```

确认数量后应用：

```bash
cargo run -- \
  --codex-home /mnt/c/Users/14139/.codex \
  --path-map 'E:\code=/mnt/e/code' \
  migrate-paths --apply
```

它会修改：

- `state_5.sqlite` 的 `threads.rollout_path`
- `state_5.sqlite` 的 `threads.cwd`
- `sessions/**/*.jsonl` 第一行 `session_meta.payload.cwd`

### 3.5 修复普通会话入口

修复 `has_user_event`：

```bash
cargo run -- \
  --codex-home /mnt/c/Users/14139/.codex \
  repair-has-user-event
```

应用：

```bash
cargo run -- \
  --codex-home /mnt/c/Users/14139/.codex \
  repair-has-user-event --apply
```

补齐 `session_index.jsonl`：

```bash
cargo run -- \
  --codex-home /mnt/c/Users/14139/.codex \
  repair-session-index
```

应用：

```bash
cargo run -- \
  --codex-home /mnt/c/Users/14139/.codex \
  repair-session-index --apply
```

### 3.6 归档、恢复、删除

归档：

```bash
cargo run -- \
  --codex-home /mnt/c/Users/14139/.codex \
  archive --id <thread-id> --apply
```

恢复归档状态：

```bash
cargo run -- \
  --codex-home /mnt/c/Users/14139/.codex \
  restore --id <thread-id> --apply
```

删除到工具 trash：

```bash
cargo run -- \
  --codex-home /mnt/c/Users/14139/.codex \
  delete --id <thread-id> --apply
```

### 3.7 从备份恢复文件

```bash
cargo run -- \
  --codex-home /mnt/c/Users/14139/.codex \
  restore --manifest /path/to/manifest.json --apply
```

只恢复 manifest 中的某个文件：

```bash
cargo run -- \
  --codex-home /mnt/c/Users/14139/.codex \
  restore --manifest /path/to/manifest.json --file /original/path --apply
```

### 3.8 校验

```bash
cargo run -- \
  --codex-home /mnt/c/Users/14139/.codex \
  validate
```

如果发现问题，命令会以非零退出码结束，方便脚本捕获。

## 4. 推荐迁移流程

当你从 Windows Codex 切到 WSL，或从默认 provider 切到自定义 provider 时，建议按这个顺序：

1. 关闭 Codex Desktop/CLI/app-server。
2. `backup`
3. `scan`
4. dry-run `migrate-provider`
5. `migrate-provider --apply`
6. dry-run `migrate-paths`
7. `migrate-paths --apply`
8. dry-run `repair-has-user-event`
9. `repair-has-user-event --apply`
10. dry-run `repair-session-index`
11. `repair-session-index --apply`
12. `validate`
13. 打开桌面端检查列表。

示例：

```bash
cargo run -- --codex-home /mnt/c/Users/14139/.codex backup

cargo run -- \
  --codex-home /mnt/c/Users/14139/.codex \
  --provider cm \
  --model gpt-5.5 \
  --path-map 'E:\code=/mnt/e/code' \
  scan

cargo run -- --codex-home /mnt/c/Users/14139/.codex migrate-provider --from openai --to cm
cargo run -- --codex-home /mnt/c/Users/14139/.codex migrate-provider --from openai --to cm --apply

cargo run -- \
  --codex-home /mnt/c/Users/14139/.codex \
  --path-map 'E:\code=/mnt/e/code' \
  migrate-paths

cargo run -- \
  --codex-home /mnt/c/Users/14139/.codex \
  --path-map 'E:\code=/mnt/e/code' \
  migrate-paths --apply

cargo run -- --codex-home /mnt/c/Users/14139/.codex repair-has-user-event
cargo run -- --codex-home /mnt/c/Users/14139/.codex repair-has-user-event --apply

cargo run -- --codex-home /mnt/c/Users/14139/.codex repair-session-index
cargo run -- --codex-home /mnt/c/Users/14139/.codex repair-session-index --apply

cargo run -- --codex-home /mnt/c/Users/14139/.codex validate
```

## 5. 常见问题

### 5.1 cannot inspect running processes on this platform

旧版本在没有 `/proc` 的平台会直接失败。现在会优先读 `/proc`，没有 `/proc` 时会退到平台进程命令检测。

如果仍然失败，请确认当前环境能执行：

```bash
ps -eo args=
```

Windows 侧则需要能执行：

```powershell
powershell.exe -NoProfile -ExecutionPolicy Bypass -Command "Get-CimInstance Win32_Process"
```

### 5.2 SQLite unable to open database file

常见原因：

- `--codex-home` 指错了。
- 当前 WSL 用户无法访问 Windows 路径。
- Codex 正在占用文件。

处理方式：

1. 确认 `state_5.sqlite` 存在。
2. 确认路径在当前 shell 可读。
3. 关闭 Codex 后重试。

### 5.3 前端构建提示 WSL 1 is not supported

这是 Node 环境问题，不是项目代码问题。可以：

- 升级到 WSL 2。
- 或从 Windows PowerShell / cmd 运行前端命令。

例如：

```powershell
powershell.exe -NoProfile -ExecutionPolicy Bypass -Command "Set-Location 'E:\code\codexSessionManager\ui'; npm run build"
```

### 5.4 Tauri cargo check 下载失败

如果看到：

```text
failed to download from https://index.crates.io/...
```

说明 crates.io 网络不可用或超时。恢复网络或提前缓存依赖后重新运行：

```bash
cargo check --manifest-path src-tauri/Cargo.toml
```
