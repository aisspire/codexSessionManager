# Codex Session Manager 使用说明

这份说明面向桌面端日常使用，重点说明如何操作、适合什么场景，以及哪些地方需要谨慎。

## 1. 使用前准备

### 1.1 确认 Codex 主目录

Codex Session Manager 需要读取 Codex 的本地会话目录。常见位置如下：

```text
Windows: C:\Users\<用户名>\.codex
WSL:     /mnt/c/Users/<用户名>/.codex
Linux:   ~/.codex
```

打开应用后，先在顶部确认“Codex 主目录”。如果目录填错，列表可能为空，或者操作的不是你想管理的那份会话数据。

### 1.2 写入前关闭 Codex

归档、活动、删除、置顶、修改会话信息之前，建议先关闭正在使用同一份数据的 Codex。

应用会在写入前做运行状态检查。如果检测到 Codex 可能正在使用同一份本地数据，会拒绝写入，避免两个程序同时修改同一份状态。

### 1.3 自行备份重要数据

当前版本暂时不提供通用的备份恢复界面。数据库修复在应用写入前会自动备份关键文件，但其他大批量整理或删除操作前，如果会话很重要，仍建议先自行复制保存 `.codex` 目录中的关键数据，尤其是：

```text
state_5.sqlite
session_index.jsonl
sessions/
archived_sessions/
```

## 2. 界面概览

桌面端主要由三部分组成：

- 左侧导航：在“批量编辑”、“会话管理”和“数据库修复”之间切换，并提供 GitHub 仓库入口。
- 顶部筛选栏：设置 Codex 主目录，按项目、提供方、模型、来源、归档状态和关键字筛选。
- 会话区域：按项目分组展示会话，支持勾选、展开分组和打开详情。

点击会话行会打开详情面板。详情面板中可以查看当前会话信息，并对单个会话执行整理操作。

## 3. 刷新和浏览会话

### 3.1 刷新列表

确认 Codex 主目录后，点击“刷新”。应用会重新读取本地会话，并按项目分组显示。

应用展示的是合并视图，不是单纯的 SQLite threads 表。它会综合读取：

- `state_5.sqlite` 中的 threads
- `sessions/**/*.jsonl` 中带 `session_meta` 的会话
- `archived_sessions/*.jsonl` 中带 `session_meta` 的会话
- `session_index.jsonl` 中的标题和更新时间

因此软件显示数量可能大于你直接查询 SQLite threads 表的数量。这通常表示有些会话只存在于 JSONL 文件中，但仍然可以被识别。

### 3.2 展开项目分组

会话会按项目分组。点击项目名前的文件夹按钮，可以展开或折叠该项目。
应用会记住项目分组的展开状态，下次打开后尽量保持同样的浏览状态。

### 3.3 查看详情

点击任意会话行，可以打开详情面板。详情中会展示：

- 会话标题
- 所属项目
- 提供方
- 模型
- 来源
- 当前状态
- 更新时间
- 会话文件路径
- 会话索引状态

详情面板适合确认单个会话是否是你要整理的目标。

## 4. 筛选和搜索

顶部筛选栏支持按以下条件缩小列表：

- 项目路径
- 提供方
- 模型
- 来源
- 活动、已归档或全部
- 搜索关键字

筛选只影响当前展示结果，不会修改会话数据。
建议先用筛选缩小范围，再进行多选和批量操作。

## 5. 选择会话

你可以用三种方式选择会话：

- 勾选单个会话行。
- 勾选项目分组，选择该项目下的所有可见会话。
- 筛选后逐个确认并勾选需要整理的会话。

批量操作只会作用于已勾选的会话。没有选中会话时，应用会提示你先选择会话。

## 6. 修改会话信息

详情面板中，部分展示信息可以直接编辑。

常见操作流程：

1. 点击需要修改的字段旁边的编辑按钮。
2. 输入新内容。
3. 确认输入。
4. 点击“保存”。
5. 等待列表刷新并确认结果。

修改会话信息适合处理标题不清晰、项目显示不准确、提供方显示不准确等轻量整理场景。

## 7. 归档、活动、置顶和删除

### 7.1 归档

归档适合把暂时不用、但仍想保留的会话从活动列表中收起来。
归档后，可以通过状态筛选切到“已归档”或“全部”重新找到它。

### 7.2 活动

“活动”会把已归档会话重新变为活动状态。
如果会话文件在 `archived_sessions` 中，应用会尽量把它移回 `sessions`。

活动完成后，应用会刷新 rollout 文件的修改时间。这样做是为了让 Codex 或列表刷新逻辑更容易感知这个会话已经回到活动状态。

### 7.3 置顶

置顶会刷新选中会话 rollout 文件的访问时间和修改时间。
它不会重写会话正文，也不会修改标题或项目，只是帮助会话更容易出现在较新的位置。

### 7.4 删除

删除会把会话移动到工具自己的回收区，不会直接永久删除。

删除前建议确认：

- 会话是否已经不再需要。
- 当前选中的会话数量是否符合预期。
- 是否已经自行保存重要数据。

## 8. 批量整理

批量整理适合一次处理多个会话，例如同一项目下的一组旧会话。

推荐流程：

1. 用筛选条件缩小范围。
2. 展开目标项目分组。
3. 勾选需要处理的会话。
4. 执行归档、活动、置顶或删除。
5. 刷新列表并确认结果。

批量操作前，尤其要注意当前勾选数量。列表中显示的选择状态是判断操作范围的主要依据。

## 9. 数据库修复

数据库修复适合在列表和实际数据不一致时使用，例如：

- 某些会话只有 JSONL 文件，但 SQLite `threads` 中没有对应行。
- SQLite 中的 `rollout_path` 为空、指向不存在的文件，或是当前系统不可用的 `/mnt/<盘符>/...` 路径。
- 会话文件已经位于 `archived_sessions`，但 SQLite 中仍显示为活动，或反过来。
- SQLite 中存在会话行，但找不到对应 JSONL 文件，需要先看清楚风险。

### 9.1 先预览，不直接修复

进入左侧“数据库修复”后，先确认 Codex 主目录，然后点击“预览修复项”。

应用会扫描：

- `state_5.sqlite`
- `sessions/**/*.jsonl`
- `archived_sessions/*.jsonl`
- `session_index.jsonl`

预览表格会显示修复类型、会话 ID、当前值、目标值和状态。默认不会写入任何文件。

### 9.2 勾选后应用

确认预览结果后，可以勾选需要修复的项目。也可以点击“全选可修复”，一次选中所有可保守处理的项目。

点击“应用已选修复”前，建议再次确认：

- 选中的项目数量是否符合预期。
- 当前 Codex 主目录是否正确。
- Codex 是否已经关闭。

应用写入前会自动检查 Codex 是否正在运行；如果检测到 Codex 可能正在使用同一份数据，会拒绝写入。

### 9.3 会自动处理的项目

数据库修复只应用可确认的保守修复：

- JSONL-only 会话：补充缺失的 SQLite `threads` 行。
- `rollout_path` 修复：把空路径、不存在路径或不可用路径修为当前真实 JSONL 路径。
- 路径归一化：把 SQLite 中不可用的 `/mnt/<盘符>/...` 类 `rollout_path` 修为当前系统下可用的真实 JSONL 路径。
- 归档状态同步：唯一 JSONL 位于 `archived_sessions` 时同步为已归档，位于 `sessions` 时同步为活动。

### 9.4 只报告、不自动处理的项目

以下情况不会自动修改：

- SQLite-only row：SQLite 中有会话行，但没有找到唯一对应 JSONL 文件。
- 重复 JSONL：同一会话 ID 对应多个 JSONL 文件。

这些项目需要人工确认，不建议让工具自动删除或猜测选择哪一个文件。

### 9.5 自动备份和边界

应用修复前会在 `<CodexHome>/backups/codex-session-manager-<timestamp>` 下备份关键文件，至少包括：

```text
state_5.sqlite
session_index.jsonl
```

如果同目录下存在 `state_5.sqlite-wal` 和 `state_5.sqlite-shm`，也会尽量一并备份。

数据库修复不会重写会话正文 JSONL，也不会删除 SQLite-only 行。它只更新 SQLite 中必要的索引字段。

## 10. 路径兼容说明

有些 Codex 数据来自 WSL 或旧环境，SQLite 中的 `rollout_path` 可能是这样的格式：

```text
/mnt/c/Users/14139/.codex/sessions/...
/mnt/e/code/project/...
```

Windows 版桌面应用在执行归档、活动、置顶等文件操作时，会把 `/mnt/<盘符>/...` 自动转换为对应 Windows 路径，例如：

```text
/mnt/c/Users/14139/.codex/sessions/... -> C:\Users\14139\.codex\sessions\...
/mnt/e/code/project/...                -> E:\code\project\...
```

这可以避免文件被误放到当前盘下的 `\mnt\c\...` 目录。

## 11. 常见问题

### 11.1 列表为空

优先检查 Codex 主目录是否正确。然后清空筛选条件，状态选择“全部”，再刷新一次。

### 11.2 软件显示数量比 SQLite threads 多

这是正常现象。应用会合并 SQLite threads 和 rollout JSONL 会话文件。
如果某些会话只存在于 JSONL 中，不在 SQLite threads 中，应用仍会尽量显示出来。

如果希望把这些 JSONL-only 会话补进 SQLite，可以进入“数据库修复”先预览，再勾选对应项目应用修复。

### 11.3 操作被拒绝

通常是因为 Codex 正在使用同一份本地数据。关闭 Codex 后再重试。
如果仍然失败，请查看应用状态栏中的错误信息。错误信息会保留失败原因，便于定位问题。

### 11.4 活动后 Codex 仍看不到会话

可以先确认：

- 当前 Codex 主目录是否正确。
- 状态筛选是否包含活动会话。
- 会话文件是否确实位于 `<CodexHome>/sessions` 下。
- SQLite 中的 `rollout_path` 是否是当前系统可识别的路径。

如果历史数据中有 `/mnt/<盘符>/...` 路径，当前版本会在文件操作时自动转换为 Windows 路径。也可以使用“数据库修复”预览并修正 SQLite 中不可用的 `rollout_path`。

### 11.5 数据库修复会改写会话正文吗

不会。数据库修复不会重写会话正文 JSONL，也不会删除 SQLite-only 行。它只补充或更新 SQLite 中可以保守确认的索引字段。

### 11.6 删除后还能恢复吗

删除不是直接永久删除，会先进入工具的回收区。
但当前版本尚未提供完整的回收区恢复界面，因此删除前仍应谨慎确认。

## 12. 使用建议

- 不确定时先刷新，再确认筛选条件。
- 大批量操作前先自行保存重要数据；使用数据库修复时，也要先查看预览结果再勾选应用。
- 删除前再看一眼选中数量。
- 发现列表和预期不一致时，不要连续操作，先检查 Codex 主目录、筛选条件和会话文件位置；必要时进入“数据库修复”查看 SQLite 与 JSONL 的不一致项。
